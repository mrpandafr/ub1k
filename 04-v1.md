# 04 — V1: Messages as Documents

> *First attempt. Good idea, wrong granularity.*

---

## 4.1 The concept

Import each SQLite message as a standalone Elasticsearch document. Add an embedding (bge-small, 384d). Query by kNN.

```
state.db → 71,175 ES documents → kNN search
```

| Metric | V1 |
|:-------|:--:|
| Documents | 71,175 |
| kNN latency | ~30ms |
| Relevance | ~50% |

---

## 4.2 The problem: dilution

A 5,000-character message might contain:

```
"Pierre Biguinet était le parrain de Nicolas. Il est mort seul à Shanghai.
Sarah, le cœur de l'Atelier. Le kill switch c'est 140k€. Umi marche sur le balcon."
```

Four distinct concepts: Pierre, Sarah, kill switch, Umi. One embedding averages them all.

The result: a query for "Sarah" returns a vector that's 25% Pierre, 25% Sarah, 25% kill switch, 25% Umi. Relevant, but polluted. 50% relevance — better than SQLite, but not good enough.

---

## 4.3 The lesson

**Granularity is not negotiable.** A memory system that stores large documents dilutes every concept into every other concept. The only fix is to shrink the document until each embedding points to exactly one idea.

This is the difference between V1 and V2. Same pipeline, different atom size.
